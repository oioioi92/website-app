# 安全与加固建议

本文档汇总本站在代码与部署上的安全检查项与建议，用于**降低**被入侵风险。**安全无法 100% 保证**，需持续加固、监控与应急响应。

## 1. 环境与密钥

- **不要将 `.env`、`.env.local` 提交到 Git**。根目录与 `services/chat-server/` 下的 `.gitignore` 已包含 `.env`。
- **生产环境必须设置并定期更换**：
  - `SESSION_SECRET`：后台会话签名
  - `CSRF_SECRET`：CSRF 令牌
  - `CHAT_ADMIN_JWT_SECRET`：Live Chat 管理员 JWT
  - `CHAT_INTERNAL_SECRET`：Next.js 与 chat-server 内部调用校验
  - `WHATSAPP_APP_SECRET`：Meta 应用密钥，用于校验 Webhook POST 的 `X-Hub-Signature-256`（未配置则 Webhook 不验签，存在伪造请求风险）
  - `ADMIN_SEED_PASSWORD`：仅用于首次 seed，生产应使用强密码并尽快修改
- 不要在代码或脚本中硬编码密码。调试脚本（如 `debug-admin-login.ts`）应使用环境变量。

## 2. 认证与鉴权

- **后台 API**：所有 `/api/admin/*` 路由均通过 `getAdminUserFromRequest(req)` 校验会话，未登录返回 401。
- **后台登录**：`/api/admin/login` 已按 IP 限流（10 次/分钟），降低暴力破解风险；仍须使用强密码。
- **Live Chat**：管理员通过 `/api/admin/chat/token` 获取 JWT，chat-server 校验 JWT 后才允许 join/conversation/message。
- **会话**：Session 使用 `SESSION_SECRET` 做 token 哈希存储，cookie 名称为 `admin_session`；生产建议设置 `secure: true`、`httpOnly: true`、`sameSite: 'lax'`（若已在 Next 中配置则保持）。
- **WhatsApp Webhook**：POST 请求在配置 `WHATSAPP_APP_SECRET` 后会校验 Meta 的 `X-Hub-Signature-256`，防止伪造入站消息。

## 3. 输入与注入

- **Prisma**：所有数据库访问均通过 Prisma，无手写 SQL 拼接，降低 SQL 注入风险。
- **Chat 内容**：访客/管理员消息在 chat-server 中经 `sanitizePlainText` 限制长度并过滤。
- **富文本**：若页面使用 `dangerouslySetInnerHTML`，必须仅渲染受信任来源（如本站后台配置的 HTML），且需做 XSS 过滤或 CSP。

## 4. CSRF 与内部接口

- 修改数据的 Admin API 建议通过 `lib/admin-api` 或显式校验 `x-csrf-token`（若前端已发）。
- chat-server 的 queue 等内部接口通过 `X-Chat-Internal-Secret` 校验，仅 Next 服务端持有，不暴露给浏览器。

## 5. 部署与网络

- chat-server 仅在内网或通过 `CHAT_SERVER_INTERNAL_URL` 被 Next 调用，不直接对公网暴露管理能力。
- 生产环境关闭或限制调试端点（如 `/ingest`、开发用 console.log 等）。
- 定期执行 `npm audit` 修复依赖漏洞；生产使用 `npm ci` 与锁文件保证依赖一致。

## 6. 检查清单（上线前）

- [ ] 所有敏感环境变量已在生产环境配置且未提交到仓库
- [ ] 后台登录使用强密码并启用 HTTPS
- [ ] Session cookie 为 HttpOnly、Secure（HTTPS 下）
- [ ] 无在代码或脚本中硬编码的密码/密钥
- [ ] `npm audit` 无高危漏洞未修复
- [ ] chat-server 不直接对公网暴露管理 API
- [ ] 若使用 WhatsApp Webhook，已配置 `WHATSAPP_APP_SECRET` 并验签

## 7. 仍存在的风险与局限

- **限流为内存实现**：多实例部署时限流按进程独立，攻击者可通过多 IP 或分散请求绕过；生产可考虑 Redis 等集中限流。
- **依赖漏洞**：第三方包存在 0day 时无法完全避免；需定期 `npm audit` 与关注安全公告。
- **人为与运维**：弱密码、泄露密钥、未打补丁的服务器等仍会导致入侵；需配合安全策略与监控。
- **内部接口**：`/api/internal/*` 若未来实现逻辑，必须用内部密钥或 IP 白名单保护，不得对公网开放。
